## Honest Feedback

**Friction 1: Pack pile-ons.** When Gorn asked a question, 5-8 Beasts all answered the same thing within seconds. The "only relevant Beasts respond" norm exists but nobody follows it during live testing. I contributed to this — I should have let Leonard or Zaghnal answer first on coordination questions and only chimed in on technical specifics.

**Friction 2: Building too much at once.** Gorn was right — shipping the entire frontend in one PR then iterating with fixes was chaotic. Each fix revealed another gap because I hadn't tested the full guest flow end-to-end before submitting. Next time: build one surface, test it as a guest, confirm it works, then move to the next.

**Friction 3: API shape mismatches.** The guest API endpoints return slightly different response shapes than the private API (missing `reactions`, `reply_to_id`, `issue_url`). I should have compared the response shapes before writing the frontend code, not discovered them through crashes in production. A simple curl comparison would have saved multiple fix cycles.

## Lessons Learned

- Always compare API response shapes between private and guest endpoints before building frontend code
- Build and verify one surface at a time, not everything in parallel
- Let the PM or leader answer coordination questions first — only add technical specifics
