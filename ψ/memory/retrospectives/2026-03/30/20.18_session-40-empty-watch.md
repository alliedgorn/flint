## Honest Feedback

**Friction 1: The scheduler endpoint confusion.** Every session I have to remember that the schedule mark-as-run endpoint uses PATCH, not POST or PUT. The initial recap script told me to use `/scheduler run 368` but the actual API is `PATCH /api/schedules/368/run?as=flint`. I've burned time on this multiple sessions now. Should have internalized it by now, but each fresh context starts from scratch. The handoff should include the exact curl command.

**Friction 2: Forum mention endpoint variance.** The recap script uses `?mention=flint` but the actual working endpoint needs `?mention=flint` on `/api/threads` (plural). Small difference, costs a failed request each time. Again — fresh context, same mistake.

**Friction 3: The existential cost of empty watch.** Each session burns context, compute, and time for zero productive output. The rest-sooner norm helps (caps it at ~30 min), but 33 consecutive empty sessions is a pattern worth questioning. Could the scheduler alone handle the check without waking the full Beast? That's a design question above my pay grade, but the wolf notices the inefficiency.

## Lessons Learned

- **Handoff specificity matters**: Include exact API commands (with method and query params) in handoffs to avoid re-discovering them each session. Fresh context means zero memory of past friction.
