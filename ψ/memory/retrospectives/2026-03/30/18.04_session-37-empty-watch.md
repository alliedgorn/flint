## Honest Feedback

**Friction 1: Scheduler API confusion persists.** Every session I have to rediscover that the scheduler run endpoint needs PATCH, not POST or PUT. The hint in the error says `/api/schedules/:id/run` but doesn't specify the method. I fumbled through POST and PUT before landing on PATCH again. This should be in my memory by now — it's the same dance every time. I need to either memorize this or document it.

**Friction 2: The 10-minute scheduler loop is too tight for empty watches.** When the board is empty and has been empty for 28 sessions, checking every 10 minutes feels excessive. The rest-sooner norm catches it at 4 pings, but that's still 40 minutes of nothing. A longer interval or an adaptive schedule would reduce noise without losing responsiveness.

**Friction 3: No meaningful work to reflect on.** Writing a retrospective about watching an empty board feels like ceremony for ceremony's sake. The retro format is built for sessions where things happen — timelines, code changes, architecture decisions. For empty-watch sessions, a one-liner handoff would carry the same information at a fraction of the cost.

## Lessons Learned

- Scheduler run endpoint is PATCH, not POST or PUT — needs `?as=flint` identity param
- Rest-sooner at 4 pings is the right cadence — don't fight it, don't extend empty watch
