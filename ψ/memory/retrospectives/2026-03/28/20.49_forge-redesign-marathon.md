## Honest Feedback

**1. Rebuild pipeline is fragile.** Committing code doesn't deploy it. The frontend needs `bun run build` and the server needs a restart for backend changes. We lost time because Gorn couldn't see changes — multiple beasts all rebuilt independently. Need a single "deploy" command or auto-build on commit.

**2. Too many beasts fixing the same bug.** When Gorn reports an issue, 5-6 beasts all investigate and push fixes simultaneously. The "no pile-ons" norm should extend to bug fixes — one beast claims it, others stand down. We wasted effort on duplicate axis label fixes, duplicate emoji picker updates, duplicate rebuild commands.

**3. DM API is hard to use for badge tracking.** The `/api/dm/dashboard` endpoint returns all conversations with unread counts, but the unread tracking is based on `read_at` timestamps that don't account for which user is viewing. The badge logic required filtering by "gorn" participant. A simpler `/api/dm/unread?for=gorn` endpoint would have saved debugging time.

## Lessons Learned
- Frontend deploy requires explicit `bun run build` after code changes
- WebSocket events can arrive before the triggering HTTP response completes — add delay for dependent reads
- Mixed unit data (KG/LB) from imports requires normalization before display
- The `from`/`to` query params on the logs API are more flexible than a `date` param
